# DESIGN.md — usage-metrics-svc スケーリング設計

実装済みの構成を前提に、「データが 100 倍になったらどう凌ぐか」を 1 枚で整理する。
**実装は増やさない。判断の解像度だけ残す。盛らない。**

## 現状（今）

- 構成: Go（読み取り専用） + Postgres の `"AILog"` + `usage_daily_rollup` + 非同期 worker。
- ベンチ実測（200k 行 / 20 テナント, Neon pg17, `EXPLAIN ANALYZE`）:
  **raw 41ms → index 5.3ms → rollup 0.05ms**（README 参照）。
- スケール特性:
  - raw / index は **テナントの行数に比例**して重くなる。
  - rollup 読みは **日数 × 次元数でほぼ一定（フラット）** — 生ログが増えても読みは変わらない。
- 読みコストは rollup でほぼ解決済み。残る変数は **worker の集計コスト**。

## データ 100 倍シナリオ（200k → 20M → 200M 行）

| 行数 | raw スキャン | index 経由 | rollup 読み | 効く対策 |
|---|---|---|---|---|
| 200k（今） | 41ms | 5.3ms | 0.05ms | — |
| 20M | 数百ms〜秒（非現実的） | テナント行数次第で悪化 | ほぼ不変 | worker を直近 N 日の**差分更新**に限定（実装済: `ROLLUP_SINCE_DAYS`）。過去日は不変なので再集計しない |
| 200M | 不可 | 不可 | rollup 自体は小さいまま | ① 生ログ集計を「前回処理の高水位 `createdAt` 以降」に絞る ② `"AILog"` を**月次パーティション**化し古い日をスキップ |

要点: 読み（API）は rollup で頭打ちにできる。100 倍で効いてくるのは worker 側なので、
**全期間の再集計をやめ、増分（直近 N 日 / 高水位）に寄せる**のが一次対策。生ログ本体には触らない
（read-only 原則を維持）。

## p95 の合成問題（現状の正直な限界）

- rollup は **grain（org×day×provider×model×dept）単位の p95** を持つが、
  日横断や window 全体の p95 は **合成できない**（percentile は加法的でない）。
- 現状の割り切り: rollup 経路は `max(grain p95)` で**近似（上界）**、raw 経路は**厳密**。
  パリティテストで calls / tokens / cost / avg は厳密一致を確認済み、p95 だけ近似。
- 正攻法（列指向に行く前に Postgres で取れる次の一手）: grain ごとに
  **t-digest / ヒストグラム**を持たせる → 任意 window の分位点を合成可能になる。

## いつ列指向（ClickHouse / BigQuery）へ寄せるか

次が**重なってきたら**検討する。単独では早すぎる:

- **テナント横断分析**が増える（全社 KPI、provider 別コスト比較、異常検知）。
  事前集計の固定軸では足りず、自由な ad-hoc `GROUP BY` が要るようになったとき。
- **append-heavy**: `AILog` は追記のみ・更新ほぼ無し → 列指向が最も得意な形。
- **集計クエリ主体**: 少数列を大量行スキャン → カラムナ圧縮 + ベクトル化実行が効く。
- 規模感: 生ログが数億〜十億行で、かつ上記の需要が実在するとき。

寄せ方（方針のみ、実装はしない）:

- `"AILog"` は変えず、CDC かバッチで ClickHouse / BigQuery に**ミラー**する
  （本サービスの read-only 原則を壊さない）。
- 本サービスは「KPI API + 直近 rollup」を担当し続け、**重い横断分析だけ**列指向に委譲。
- **BigQuery** = 運用レス・従量課金 / **ClickHouse** = 低レイテンシ・自前運用。
  中小企業フェーズでは運用コスト最小の **BigQuery 寄り**が妥当。

## やらないこと（盛らない）

- 今の規模（数十万〜数百万行）で列指向は**過剰**。Postgres + index + rollup で十分。
- Kafka 等のストリーミング基盤やマイクロサービス分割も現時点で不要。
  worker の ticker + 冪等 upsert で足りている。
- 移行は「需要が出てから」。今は**判断基準と移行パス**を用意するに留める。
